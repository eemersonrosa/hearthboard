# Plan: public demo on GitHub Pages ("demo mode")

Status: **implemented**. Demo mode lives in `client/src/demo/` (MSW handlers +
date-relative fixtures) behind `VITE_DEMO_MODE=1`; the repo is public and the
demo deploys to https://eemersonrosa.github.io/hearthboard/ via
`.github/workflows/deploy-demo.yml` on every push that touches `client/`.
Build locally with `VITE_DEMO_MODE=1 npx vite build --outDir dist-demo`.

## Goal

A read-mostly, interactive showcase of Hearthboard at a public URL: real UI,
sample family (users, chores, schedules), a populated calendar, theming, tabs —
all running **without a backend**. Visitors can complete chores, switch tabs
and themes; state lives in browser memory and resets on reload.

## Why a mock layer (and which one)

GitHub Pages is static hosting — no Fastify, no SQLite. The client is a clean
SPA that talks to `/api/*`, so the whole backend can be simulated in-browser.

Use **MSW (Mock Service Worker)** rather than an axios adapter:

- The app uses axios (XHR) everywhere **except** `utils/timezone.js`, which uses
  `fetch` and runs in `main.jsx` *before* React renders. MSW intercepts both
  XHR and fetch at the service-worker level, so no app code changes.
- Handlers keep in-memory state, so mutations (complete/uncomplete a chore,
  reassign, move widgets) actually work during a session.
- Zero footprint in production builds: the worker + fixtures load only behind
  the demo flag and land in their own lazy chunk.

## Architecture

```
client/
├── src/demo/
│   ├── handlers.js     # MSW request handlers for the /api surface
│   ├── fixtures.js     # Sample data generators (dates relative to "today")
│   └── browser.js      # worker setup (msw/browser)
├── public/mockServiceWorker.js   # generated by `npx msw init client/public`
└── vite.config.js      # base: DEMO ? '/<pages-path>/' : '/'
```

Bootstrapping in `main.jsx` (the only app-code touch):

```js
if (import.meta.env.VITE_DEMO_MODE === '1') {
  const { worker } = await import('./demo/browser.js');
  await worker.start({ serviceWorker: { url: import.meta.env.BASE_URL + 'mockServiceWorker.js' } });
}
// existing initTimezone().finally(render) stays as-is — the worker must be
// started first so /api/timezone is already interceptable.
```

Build with `VITE_DEMO_MODE=1 VITE_REACT_APP_API_URL='' npm run build`.

## Endpoints to mock (minimum viable demo)

From the dashboard's load path — keep handlers small, return the same shapes as
`docs/reference/backend-api.md`:

| Area | Endpoints | Behavior |
| --- | --- | --- |
| Boot | `/api/timezone`, `/api/stats`, `/api/admin-pin/exists` | Static. No PIN so visitors can open Settings. |
| Device | `/api/devices/:name/settings` (GET/PUT/PATCH), `/api/devices`, `/api/devices/:name/tabs`, `/api/devices/:name/widget-assignments*` | Seeded from fixtures; mutations update in-memory maps so drag/resize works. |
| Users & chores | `/api/users`, `/api/chores*`, `/api/chore-schedules*`, `/api/chore-history*`, `/api/chores/complete`, `/api/chores/uncomplete` | 3 sample users, ~8 chores (daily/weekly/sticky/bonus mix). Complete/uncomplete mutate in-memory history. |
| Calendar | `/api/calendar-events`, `/api/calendar-sources`, `/api/calendar-sync/status` | 1 fake "Family" source; ~20 events generated **relative to today** (birthdays, practice, appointments, a multi-day trip) so the demo never looks stale. |
| Weather | `/api/settings`, `/api/settings/search`, `/api/homeassistant/weather` | Route weather through the HA source with a canned forecast — avoids needing an OpenWeatherMap key in the demo. |
| Photos | `/api/photo-sources`, `/api/photo-items` | Optional v2: a "Hearthboard uploads" source with 5–6 bundled royalty-free images in `src/demo/photos/`. Skip in v1 (don't enable the widget). |
| Sounds/HA/plugins | `/api/sounds`, `/api/homeassistant/status`, `/api/widgets` | Empty/disabled responses so their admin tabs render gracefully. |
| Backup | `/api/config/export`, `/api/config/import` | Export works from fixture state (nice showcase); import returns a friendly "disabled in demo" error. |

Catch-all `/api/*` handler returning `501 { error: 'Not available in demo' }`
so anything unmocked degrades visibly instead of hanging.

## Fixture design notes

- **Dates are generated, not hardcoded** — `fixtures.js` builds everything off
  `new Date()` so chores are always due "today" and the calendar month is
  always populated.
- Seed one device profile (the shared default) with 3 tabs: Home (chores +
  calendar week), Calendar (month view), plus an empty tab to show tab editing.
- A "Demo" ribbon/banner (tiny conditional in `app.jsx` behind the same flag,
  or injected via a fixture-driven HA alert) telling visitors data resets on
  reload.

## GitHub Pages specifics

- `base` in `vite.config.js` must match the Pages path (e.g.
  `/hearthboard-demo/`); use `import.meta.env.BASE_URL` for the worker URL.
- The `/photos` path check in `main.jsx` compares full `pathname`; under a
  subpath it simply falls through to the dashboard — fine for the demo.
- Copy `index.html` → `404.html` in the deploy step (SPA fallback).
- MSW requires HTTPS or localhost — Pages is HTTPS ✓.

## Verification

- `VITE_DEMO_MODE=1 npm run dev` locally: dashboard loads with sample data and
  **zero** requests leaving the browser (check devtools network tab).
- `npm run build` (without the flag) must not grow the bundle or register the
  worker — demo code stays in a lazy chunk that never loads.
- Existing vitest suites unaffected.

## Publishing (decision pending)

The repo is **private**; Pages on private repos needs a paid plan, and a demo
page is public regardless (NOTICE currently says "private and not
distributed" — publishing the built frontend is distribution under AGPL, which
is permitted but changes that stance).

- **Option A — separate public repo (recommended):** create
  `eemersonrosa/hearthboard-demo`; a workflow in this repo builds the demo
  bundle and force-pushes `client/dist` to that repo's `gh-pages` (via a
  fine-grained PAT or deploy key stored as a secret). Source stays private;
  only compiled assets + fixtures become public.
- **Option B — make this repo public:** enable Pages (Source: GitHub Actions)
  and add a `deploy-demo.yml` using `actions/upload-pages-artifact` +
  `actions/deploy-pages`. Requires updating NOTICE/README wording first.
